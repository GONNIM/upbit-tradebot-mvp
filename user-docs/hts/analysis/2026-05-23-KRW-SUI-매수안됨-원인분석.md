ID 894 | 2026-05-23 07:58 | BUY  | EMA_GC              | price=1586 |  ← 봇 BUY

이는 **포지션 라이프사이클의 명백한 위반**이다 — 매도된 BUY 가격이 후속 매도에 재사용되어서는 안 된다.

## 4. 봇이 매수 안 한 이유 (사용자 클레임 본질)

### 4-1. EMA 전략 buy 조건 (`mcmax33_EMA_buy_sell_conditions.json`)

```json
"buy": {
  "ema_gc": true,                  ← Golden Cross 활성
  "above_base_ema": false,
  "bullish_candle": false,
  "surge_filter_enabled": true,
  "surge_threshold_pct": 0.01      ← 1% (매우 타이트!)
}
```

### 4-2. 22:00~23:00 봇 동작

- 22:00 ema_fast(30)=1504.28 < ema_slow(200)=1507.50 → **Dead 상태**
- 22:55~23:00 ema_fast=1526~1527 > ema_slow=1514~1515 → **Golden 정렬** (gap +12)
- → 그 사이(22:00~22:54) 어딘가에서 **Dead → Golden Cross 이벤트 발생했음**이 확실

### 4-3. 그러나 audit_buy_eval에는 22:00~23:42 매수평가의 `cross_status`/`ema` 값이 모두 NULL

→ **이 시간대 buy_eval 기록이 정상 평가가 아니다.** 추후 23:40:50 봇 재시작 시 WARMUP_IN_PROGRESS 일괄 기록(`완료 1/200, 2/200…`)이 들어간 흔적과 결합하면, **실시간 평가가 누락된 봉이 다수 존재**한다는 의미.

### 4-4. REST API Reconcile 실패 누적

```
May 23 18:07~20:06 ERROR core.rest_reconcile | 최대 재시도 초과 (5회, 25초 대기) | → 봉 스킵
```

18:00~20:06 사이 Reconcile 실패 13건 이상 → 봉 데이터 결손. 이게 후속 평가에 영향.

### 4-5. 결론 — 매수 미발동 원인

봇이 매수 안 한 이유는 **두 가지 복합**:
1. **봉 데이터 누락 + 평가 사이클 결손** → Golden Cross "이벤트" 봉이 실시간 평가되지 않았을 가능성 (Cross는 봉 단위 prev/curr 비교라 한 번 놓치면 영구 미감지)
2. `surge_threshold_pct: 0.01` (1%) 매우 타이트 → 22:55 시점 가격 1,530 vs ema_slow 1,515 → **이격 0.99%**, Cross가 감지됐어도 SURGE_FILTER로 즉시 차단 가능 범위

## 6. 조치 권고 (CTO)

### 즉시 조치 (24시간 내)

1. **POSITION-SYNC fallback 차단**
   - `strategy_engine.py`의 "DB에서 last BUY 가져와 진입가 자동 복구" 로직을 **즉시 비활성화**
   - 외부 매수가 감지되었으나 정확한 진입가를 모를 경우 → **SELL 발동 금지** (HOLD 유지)
   - 사용자에게 알림 → 수동으로 진입가 확정 요구

2. **bars_held 음수 발생 시 SELL 강제 차단**
   - `core/strategy_incremental.py`의 `[EMA] bars_held=음수 감지 → 보정` 로직 제거
   - 음수 발생 = 데이터 무결성 깨짐 신호 → **SELL 즉시 차단 + 에러 알림**

### 단기 조치 (1주 내)

3. **HTS_BUY 감지 보완**
   - 사용자가 봇 운용 중에 수동 매수했을 때 `prev_qty==0` 조건이 깨지는 케이스 검증
   - Upbit API의 `avg_buy_price`를 신뢰 가능한 단일 진입가 소스로 사용

4. **REST Reconcile 실패 회복 강화**
   - 5회 재시도 실패 시 단순 "스킵"이 아닌 **백오프 + 즉시 알람**
   - 데이터 결손 봉이 연속 N개 이상이면 자동 정지

5. **audit_buy_eval 기록 누락 디버깅**
   - 22:00~23:42 cross_status/ema 필드 NULL 원인 추적
   - upsert 로직과 BACKFILL 평가 모드의 상호작용 검증

### 중기 (2주)

6. **포지션 라이프사이클 무결성 보장**
   - audit_trades 상에서 매도된 BUY는 "closed" 마킹
   - POSITION-SYNC가 closed BUY를 참조 못 하도록 격리

7. **외부 매수 자동 매매 정책 결정**
   - 봇이 외부 매수를 자동 인수해서 매도 관리할지 / 무시할지 정책 명확화
   - 현재처럼 "감지하면 무조건 흡수해서 STOP_LOSS 적용"은 위험

## 7. 사용자 클레임 답변 (최종)

**Q. "Golden Cross 발생했는데 왜 매수가 안 됐나?"**

A. 두 가지가 동시에 일어났다:

1. **봇이 매수를 못 한 이유**: 22:00 ~ 23:00 동안 REST 데이터 결손 + 매수평가 audit 누락으로 **Golden Cross 이벤트 봉 자체를 실시간 평가하지 못했을 가능성이 매우 높음**. EMA Cross는 봉 단위 prev/curr 비교라 한 번 놓치면 다시 재정렬되기 전까지 영구 미감지. 추가로 `surge_threshold_pct=1%`가 매우 타이트해서 Cross가 감지됐어도 SURGE_FILTER에 막혔을 가능성도 있음.

2. **그 와중에 사용자 보유분이 강제 매도된 사건**: 23:01에 봇이 사용자 지갑(633.638 SUI)을 감지하고도 진입가를 모르자, DB의 14시간 전 옛 BUY 가격(1,586)을 **임의 차용**해서 PNL -3.59% 계산 → STOP_LOSS 강제 발동. **이건 시스템 결함이며 즉시 조치 필요.**
